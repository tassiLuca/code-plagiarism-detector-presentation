Fase 1: Analisi

flowchart TB subgraph analysis direction LR Parsing[1.Parsing] -- AST --> Preprocess[2.Preprocessing] Preprocess -- Cleaned AST --> Tokenization[3.Tokenization] end
  1. I file sono parsati $\rightarrow$ è generato l’Abstract Syntax Tree
  2. Preprocessing: l’AST viene “ripulito” da dettagli insignificanti a livello semantico (commenti, dichiarazioni di import, package, …)
  3. L’AST viene visitato e vengono “emessi” i token corrispondenti ai costrutti del linguaggio da valorizzare
    • ⚠️ la scelta dei tipi di token è un fattore critico

Fase 2: Filtraggio

flowchart TB subgraph filtering[filtering] direction LR Indexing[4.Indexing] -- Index DS --> Filtering[5.Filtering] end
  • i token sono indicizzati in coppie chiave-valore
    • chiavi: tipi di token
    • valore: frequenza (numero di occorrenze all’interno del sorgente)
  • similarità coseno
    • tutte le coppie di rappresentazioni la cui similarità è inferiore a un valore di soglia sono escluse: $threshold = sim_{min} + \text{init_threshold} \cdot (sim_{max} - sim_{min})$

Fase 3: Detection

  • È applicato un algoritmo di string matching (RKR-GST, 1993), riadattato per funzionare su sequenze di token
  • ⚠️ fase più onerosa computazionalmente $\Rightarrow$ esecuzione parallela
    • RKR-GST ha complessità, nel caso pessimo, $O(n^2)$

Metriche

Come vengono stimate le similarità?

Similarità tra coppie di sorgenti

Data una coppia di sorgenti $A$ e $B$:

$$ \text{max_sim_s}(A, B) = \frac{\sum_{match \in tiles} length}{min(|A|,|B|)} $$

⚠️

$|A|=30, |B|=150, \sum_{match \in tiles} length = 30$

$\Rightarrow \text{max_sim_s(A,B)}=1$

Data una coppia di sorgenti $A$ e $B$:

$$ \text{avg_sim_s}(A, B) = \frac{2 \cdot \sum_{match \in tiles} length}{|A|+|B|} $$

✅ $|A|=30, |B|=150, \sum_{match \in tiles} length = 30 \Rightarrow \text{avg_sim_s(A,B)}=0.33$

$\hspace{1cm}$

⚠️

$|A|=|B|=30, \sum_{match \in tiles} length = 25 \Rightarrow \text{avg_sim_s(A,B)}=0.83$ $|A|=|B|=1000, \sum_{match \in tiles} length = 400 \Rightarrow \text{avg_sim_s(A,B)}=0.4$

Similarità tra coppie di progetti

La similarità tra due progetti, $A$ e $B$, è data da:

$$ sim_p(A, B) = max \biggl( \frac{|\text{reported sources of A}|}{|\text{sources of A}|}, \frac{|\text{reported sources of B|}}{|\text{sources of B}|} \biggl) $$

$$ \cdot P_{75}(\text{similarity of reported sources}) $$

dove $P_{75}$ è il 75° percentile (o 3° quartile) delle similarità dei sorgenti segnalati.

Il tool sviluppato

  • sviluppato ex novo in Kotlin, open source

disponibile al link git@github.com:DanySK/code-plagiarism-detector/

  • permette di reperire i progetti da repository GitHub e Bitbucket
  • genera un report testuale con la stima di similarità repo-to-repo e con le parti di codice simili
  • è configurabile:
    • lunghezza minima della sequenza di token da riportare (min-tokens)
    • soglia di similarità tra sorgenti (min-duplication $\in [0, 1]$)
    • soglia di filtraggio dei sorgenti (threshold-filter $\in [0, 1]$)
    • esclusione di file ritenuti non rilevanti/comune utilizzo (e.g. Pair.java)

compare50

progetto GitHub | documentazione

  • Harvard, 2018
  • implementa 5 tipi di confronto:
    • strutturale (default ON): tokenizzazione + Winnowing
    • testuale (default ON): rimozione degli spazi + Winnowing
    • letterale (default ON): Winnowing
    • senza commenti (default OFF): rimuove i commenti (ma tiene gli spazi) + Winnowing
    • 😯 “misspellings” (default OFF): confronta i commenti per parole inglesi con errori di ortografia identici.

ASPETTI NEGATIVI

  • i report non sono ordinati per sorgente più simile
  • non è configurabile nella soglia di token da riportare
  • sono considerati dichiarazioni irrilevanti che appesantiscono inutilmente i report
